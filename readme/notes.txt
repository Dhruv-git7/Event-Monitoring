SLIDE 3 — Problem Statement
The problem I focused on is infrastructure visibility.

In a technical environment, there are many different types of systems. Some machines may run Linux, some may run Windows, some may be virtual machines, some may be containers, and some may be network devices or application servers.

If each system is checked manually or through separate tools, it becomes difficult to understand the complete health of the infrastructure.

For example, one server may have high CPU usage, another may have low disk space, and another may have a stopped service. If there is no unified dashboard, these issues may be detected late.

So the challenge was to build a system that can continuously collect metrics, detect abnormal conditions, and show everything in one place in real time.

SLIDE 4 — Project Objective
The objective of the project was to build a working infrastructure monitoring platform from scratch.

The platform should collect system metrics from hosts using a monitoring agent. It should send those metrics to backend services, process them, store them, evaluate alert rules, and show live status on a dashboard.

The project was inspired by tools like Zabbix, Nagios, Grafana, and Datadog Infrastructure Monitoring.

The goal was not only to create a user interface, but to build the complete backend pipeline also: agent, API service, Kafka queue, metric processor, database storage, alert engine, WebSocket live delivery, and dashboard.

So the final result is a full-stack distributed monitoring system.

SLIDE 5 — Development Timeline
The project was developed in two phases during the internship.

In the first three weeks, I built a Universal Real-Time Event Ingestion Platform. This was inspired by Splunk and Elasticsearch. It could receive events, parse different formats, store them, and stream them live to the dashboard.

SLIDE 6 — Phase 1 Summary
Phase 1 was focused on universal event ingestion.

The system could receive data through HTTP and Kafka. It could parse different data formats such as JSON, CSV, XML, Syslog, and Key-Value format.

The main idea was that the system should not require a fixed schema in advance. It should be able to accept different types of events and normalize them.

This phase also implemented persistent storage, authentication using JWT and API keys, and live dashboard updates using WebSocket.

The most important outcome of Phase 1 was that it proved the architecture. It showed that data could move from source to backend to storage to dashboard in real time.

That same architectural foundation was later reused for the monitoring platform.

SLIDE 7 — Why the Pivot
After Phase 1, I moved from log/event analytics to infrastructure monitoring.

The reason was that infrastructure monitoring has a very direct operational use case. Instead of only receiving generic logs, the system could monitor actual machines and services.

Monitoring requires a more structured model. We need concepts like hosts, host groups, metrics, alert rules, problems, services, and heartbeats.

So the database schema and dashboard had to be redesigned.

The earlier event pipeline helped because it already had the concepts of ingestion, processing, storage, and live delivery. I adapted those same ideas for metric monitoring.

This shows the engineering process: first validate the distributed architecture, then specialize it for a stronger use case.

SLIDE 8 — Final System Architecture
This is the final architecture of the system.

At the left side, we have the Monitoring Agent. This is a lightweight Node.js script that runs on the monitored machine. It collects CPU, RAM, disk, network, uptime, process, and service metrics.

The agent sends this data to Agent API, which runs on port 4000. Agent API validates the agent key and publishes the metric payload to Kafka.

Kafka acts as the message queue. It decouples data ingestion from processing.

The Metrics Processor consumes messages from Kafka. It writes metrics to PostgreSQL and ClickHouse, evaluates alert rules, updates host status, updates service checks, and publishes live events to Redis.

The WebSocket Gateway subscribes to Redis and sends live updates to the browser.

Finally, the Next.js Dashboard displays host status, live gauges, charts, alert problems, and reports.

So the complete flow is:

Agent → Agent API → Kafka → Metrics Processor → PostgreSQL + ClickHouse + Redis → WebSocket Gateway → Dashboard

SLIDE 9 — Technology Stack
This slide shows the technologies used and why they were selected.

Node.js 20 was used as the runtime because both the backend services and monitoring agent could be written in the same ecosystem.

Fastify was used for REST APIs because it is lightweight and works well for service-based backend design.

Kafka was used as the message queue because metric ingestion and processing should be separated. If processing is slow, the agent API can still accept data and Kafka can buffer the messages.

PostgreSQL was used for relational data such as users, hosts, alert rules, problems, sessions, and API keys.

ClickHouse was used for time-series metric storage because historical metric queries are better handled by a column-oriented database.

Redis was used for PubSub so that processed metric updates could be pushed live to the WebSocket Gateway.

Next.js was used for the dashboard, and Recharts was used for metric charts.

SLIDE 10 — Microservices Design
The system is divided into focused services.

The first service is agent-api, running on port 4000. It receives agent registrations, heartbeats, and metric check-ins.

The second service is auth-api, running on port 4002. It handles authentication and all dashboard REST APIs such as hosts, problems, alert rules, reports, and admin functions.

The third service is metrics-processor. It is a Kafka consumer. It processes metric payloads, stores them, evaluates alert rules, and updates host status.

The fourth service is ws-gateway, running on port 4001. It sends live updates to the dashboard through Socket.io.

The fifth component is the monitoring agent, which runs on the monitored machine.

This design keeps each service focused and makes the system easier to understand, test, and scale.

SLIDE 11 — Database Design
The project uses two types of databases.

PostgreSQL is used for relational and operational data. It stores tenants, users, sessions, API keys, hosts, host groups, heartbeats, metric values, alert rules, problems, host services, and audit logs.

Metric values are partitioned monthly in PostgreSQL because monitoring data grows continuously.

ClickHouse is used for time-series analytics. It stores high-volume metric readings and supports hourly and daily rollups.

The reason for using two databases is that monitoring systems have two different types of data.

Current operational state, such as host status and problems, fits well in PostgreSQL. Historical metric analysis fits better in ClickHouse.

So PostgreSQL is used for system state, and ClickHouse is used for metric history.

SLIDE 12 — Monitoring Agent
The monitoring agent is one of the most important parts of the project.

It is a single JavaScript file written using only built-in Node.js modules. It has zero npm dependencies.

It works on both Windows and Linux.

For CPU usage, it samples os.cpus() twice with a short delay and calculates usage from idle time difference.

For RAM, it uses os.totalmem() and os.freemem().

For disk usage, it uses df -k / on Linux and wmic on Windows.

For network usage, it reads /proc/net/dev on Linux and uses netstat -e on Windows.

For service checks, it attempts TCP connections to configured ports like 80, 443, 5432, and 6379.

The agent sends heartbeat messages every 10 seconds and full metric check-ins every 30 seconds.

SLIDE 13 — Alert Engine
The alert engine works using threshold rules.

For example, an alert rule may say:

CPU usage greater than 75 percent is warning, and greater than 90 percent is critical.

When the Metrics Processor receives a metric value, it checks enabled alert rules for that host.

If the value crosses the threshold, the system opens a problem.

If the value returns to normal, the system automatically resolves the problem.

The problem workflow has three states:

active → acknowledged → resolved

Active means the issue is currently present. Acknowledged means an operator has seen it. Resolved means the issue has cleared.

This is similar to the trigger and problem concept in Zabbix.

SLIDE 14 — Real-Time Delivery
The real-time part of the system works through Redis and WebSocket.

After Metrics Processor processes a Kafka message, it publishes a live event to Redis.

The WebSocket Gateway subscribes to Redis channels like metrics:{host_id} and problems:global.

When a Redis message arrives, the gateway emits events to the browser using Socket.io.

The dashboard listens for events like:

metric_update, host_status, problem_opened, problem_resolved, and heartbeat.

This allows CPU gauges, RAM gauges, host status, and problem lists to update without refreshing the page.

During testing, the end-to-end update from agent check-in to dashboard update was observed in under two seconds.

SLIDE 15 — Dashboard Overview
The dashboard is designed like an enterprise monitoring console.

It uses a light theme, compact tables, status badges, and a fixed sidebar, similar to tools like Zabbix.

The overview page shows total hosts, online hosts, offline hosts, warning hosts, critical hosts, and active problems.

SLIDE 21 — Technical Challenges
There were several technical challenges during implementation.

One challenge was module compatibility. Some npm packages are ESM-only in newer versions, while the backend services were configured for CommonJS TypeScript. This required careful package selection, such as pinning uuid to version 8.3.2.

Another challenge was Kafka restart behavior. Kafka stores topic state in Zookeeper, so restarting during development can create topic-exists errors. I handled topic creation in an idempotent way.

The third challenge was cross-platform metric collection. Windows and Linux expose disk and network information differently, so the agent needed separate logic for each operating system.

These issues were useful because they reflected real engineering problems, not just coding syntax problems.

SLIDE 22 — Engineering Decisions
This slide explains important design choices.

Kafka was chosen instead of direct database writes because ingestion and processing should be separated.

ClickHouse was chosen in addition to PostgreSQL because historical metric queries can become heavy in monitoring systems.

Fastify was used for APIs because it is lightweight and suitable for service-based backend design.

Agent-based monitoring was chosen because it gives direct access to OS-level metrics.

Redis PubSub was used instead of dashboard polling because it allows live updates with lower latency.

Each decision was made based on the role of that component in the monitoring system.

SLIDE 24 — Comparison with Zabbix
This project is inspired by Zabbix, but it is not meant to replace Zabbix.

Zabbix is a mature production monitoring system with many years of development.

My platform implements core monitoring ideas such as host monitoring, agent-based metrics, alert rules, problems, acknowledgements, historical charts, APIs, and authentication.

However, features like SNMP monitoring, auto-discovery, templates, and large-scale production tuning are still future work.

So the comparison should be understood honestly: this project is a working prototype that demonstrates Zabbix-style architecture and workflows.

SLIDE 25 — Limitations
The current system has some limitations.

Formal production-scale load testing was not performed.

SNMP-based agentless monitoring is not implemented yet.

Email and SMS notification channels are not integrated. Webhook support is the initial direction.

Kubernetes deployment is not configured.

Network auto-discovery and SSL certificate monitoring are also not implemented.

These are not failures. They are scope boundaries because the internship duration was limited.

The important point is that the core architecture is ready for these extensions.

SLIDE 26 — Future Roadmap

How to Explain Why You Used Kafka
Kafka is used so that data ingestion and processing are separated. If the processor or database is temporarily slow, Agent API can still accept metric payloads and place them in Kafka. This makes the architecture more reliable than directly writing every check-in to the database from the API.

How to Explain Why You Used ClickHouse
PostgreSQL is good for users, hosts, alert rules, and problems. But monitoring systems generate many time-series metric rows. ClickHouse is better suited for analytical queries on large metric histories. That is why I used PostgreSQL for operational state and ClickHouse for time-series metrics.

How to Explain Why You Used Redis and WebSocket
Redis is used for internal live event publishing. WebSocket Gateway listens to Redis messages and sends them to the browser. This allows the dashboard to update live without refreshing or polling the database repeatedly.

How to Explain Alert Engine
Alert rules define thresholds for metrics. For example, CPU usage greater than 75 percent is warning and greater than 90 percent is critical. When a metric crosses the threshold, the system opens a problem. If the metric becomes normal again, the system resolves the problem automatically. Users can also acknowledge problems.

How to Explain Agent
The agent is a small Node.js script with no external dependencies. It uses built-in OS functions and system commands to collect metrics. It sends heartbeat messages every 10 seconds and full check-ins every 30 seconds. This agent-based approach gives accurate OS-level data.
